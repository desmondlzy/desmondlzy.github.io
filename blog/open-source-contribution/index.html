Why We Should Engage More in Open Source?

December 15, 2024

The idea of writing this post started pretty straightforward. I made my first code contribution to a sizeable open source project (nerfstudio-project/gsplat, 2.4k stars). It might sound like I’m bragging, but please hear me out.

The size/difficulty of the contribution is really nothing to write home about (but I’m still gonna write about it here lol). The pull request can been seen here. tl;dr: It’s a tiny 3-line fix include guard naming.

For those who are not too familiar with c/cpp/cuda programming, to avoid a header file included twice that lead to compilation errors such as redefinition, we usually add include guard macros to each header file like this.

#ifndef HEADER_FILENAME_H
#define HEADER_FILENAME_H 

code...

#endif // HEADER_FILENAME_H

This practical use of macros is pretty easy-to-understand, and is pretty much taught in most of the introductory c/cpp programming courses in a uni’s computer science curriculum. Even they look perfectly harmless, but these include guards can easily catch the programmers off guard. Namely, if you mess up HEADER_FILENAME_H, like #ifndef and #define followed by different names, or having the name clash with another file’s include guard’s, you might get hundreds of lines of compiler error messages in which nothing points to the include guard name typo (thank you, c macros, I guess). All the messages might not be too hard for seasoned devs to parse, but can be quite overwhelming for someone new just gets their hand on this.

This time the fix I proposed is no different. I spotted it while I was using their code, got the errors, investigated, and fixed it locally. But the true difference is, this time my efforts didn’t just stop there. Instead, I tried to fix it for everyone.

You see, trying to fix it for everyone isn’t hard at all: check out other pull requests’ formats, create a pull request, nicely communicate the purpose and intents with the maintainers; simple as is. The hard thing is the awareness/willingness to do so.

Even a tiny fix can potentially save hundreds of developers’ precious time and frustration – especially those who don’t know how to deal with this issue.

If you’ve never heard it, open source community has been truly doing a terrific job. The Linux kernel, and all the variety of distros, toolchains, ecosystem around it are powering most of the servers, mobile phones (android), and smart devices. Blender is yet another open source example that I frequently use in my own research without which I probably wouldn’t have made this far. It is hard to believe such all these creations that are relevant literally everyone often time are just free, not only free to use its functionality, but also “have the freedom to run, copy, distribute, study, change and improve the software.” per GNU. This was shocking to me when I first heard about it, especially when I had known was literally just you pay money to get the service from some big tech companies.

This is technically not the first open source contribution of mine. I’ve been trying to open source my research code (such as my embroidery project), and some side projects if they potentially would help people other than me. But the contribution to gsplat is the first time I really feel the difference. It’s partly because of the scale and the amount of interest in this project.

The project is gsplat. It implements a differentiable rendering algorithm called 3D Gaussian Splatting, and a series of algorithms improving it. I use it intensively in my current research project. The role it plays is so fundamental that I would say the project would have absolutely no chance of getting to the current stage without it. That’s why I have been nothing but thankful to the developers and maintainers of gsplat.
